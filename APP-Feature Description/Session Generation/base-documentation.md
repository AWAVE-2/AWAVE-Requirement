# Session Generation – Base Documentation (QWAVE / AWAVE)

This analysis is based on the technical documents, code snippets, and concept diagrams provided for the QWAVE / OWOVE – MORE THAN MEDITATION project. It describes a highly complex, interactive audio platform for guided meditations and soundscapes.

---

## 1. System Overview and Architecture

The system is designed to generate dynamic audio sessions consisting of multiple synchronized layers. The architecture distinguishes between different user roles: patients, private users, therapists, and administrators.

### Core Components of the Audio Modules

A session consists of up to **six parallel audio tracks**, subdivided into phases (Phase I to IV+):

| Module    | Description |
|----------|-------------|
| **Text** | Intros, affirmations, or closing texts (`.mp3`) |
| **Music** | Genre-based background music (e.g. Piano, ZenGarden) |
| **Nature** | Nature sounds (e.g. forest) |
| **Frequency** | Synthetically generated frequencies via oscillators (no files) |
| **Noise** | Various noise colors (White, Pink, Brown, Grey, Blue, Violet) |
| **Sound** | Accent sounds such as bells or alarms at intervals |

---

## 2. Technical Implementation (Audio Engine)

The application uses a combination of standard HTML5 `<audio>` elements and the **Web Audio API**.

### Dynamic File Path Generation (`getAudioSrc`)

The system does not load audio assets statically; it generates paths dynamically via the function `getAudioSrc(dbItem)`:

- **Object mapping:** Strings from the configuration are converted into DB objects from `content-database.js`.
- **Voice logic:** For texts, the filename is extended with the selected voice name (e.g. `../path/voicename_textname.mp3`).
- **Randomization:** For music and text pools, a random number is generated. To avoid redundancy, the same file is only repeated after at least 2/3 of the pool has been played.

### Player Logic

- **Text player:** Works in two modes: `one` (single play) or `loop` (repetition with variable delays between affirmations).
- **Music player:** Supports seamless transitions between phases. When two consecutive phases share the same genre, the music continues without interruption.
- **Nature player:** Uses a `timeupdate` event to create a seamless loop with an overlap (`overlapTime`) of 2 seconds.
- **Noise & Frequency:** Are modulated directly via the Web Audio API. A `ChannelSplitter` allows separate processing of left and right channels (e.g. for binaural beats).

---

## 3. Signal Flow Analysis (Frequency Generator)

The frequency generator documentation shows complex routing diagrams for different signal types:

- **Binaural/Monaural:** Signals are generated by root oscillators, modulated by sweep frequencies, and routed to the master output via gain nodes.
- **Sync-Noise:** Noise signals (WAV samples) are driven through filters (bandpass) and synchronized with LFOs (Low Frequency Oscillators) for rhythmic effects.

---

## 4. Session Control and UI Logic

Control is handled mainly via the file `main.js` and the UI module `editor-live.js`.

- **Session start:** The function `startSession()` triggers `startPhase(1)`, which in turn calls all individual player functions (`startTextPlayer`, `startMusicPlayer`, etc.).
- **Live mode:** During a running session the user can change content (e.g. music genre). The system decides by type whether only the affected player or the entire session is restarted.
- **Preview function:** Allows previewing voices or soundscapes before the actual start.

---

## 5. Future Concept: The Adaptive Algorithm

A significant part of the material deals with the "Session Generator Concept", which envisions three adaptation scenarios:

| Case   | Method       | Description |
|--------|---------------|--------------|
| Case 1 | Manual       | Session length is set in advance; phases adapt evenly. |
| Case 2 | Modificative | Phases can be removed or replaced during planning. |
| Case 3 | Sensor measurement | Dynamic phase correction via real-time data (EEG, RPG, REM, gyroscope). |

This concept points to a **bio-adaptive feedback loop** in which the meditation responds to the user's mental and physical state.

---

## Summary of the Analysis

QWAVE is a technologically ambitious system that goes beyond simple meditation apps. Via the Web Audio API it offers audiophile precision in frequency therapy. The algorithmic file management ensures that content remains varied even with frequent use. The core innovation lies in the planned sensor coupling to adapt the audio tracks in real time to the user's brainwaves or sleep phases.

## Content-IDs for iOS Session Playback

In the native iOS app (Swift), session phases are resolved via **Content-IDs**: `PlayerViewModel.preResolveSessionContentURLs` collects all content IDs from the phases and calls `soundRepository.getSound(byContentId: contentId)`. Firestore documents in the `sounds` collection must have a **`contentId`** field with exactly the following values (as output by `SessionGenerator`):

| Layer   | Schema / Examples | Origin in Code |
|---------|--------------------|----------------|
| **Text** | `voice/topic/stage`, e.g. `franca/sleep/intro` | `SessionGenerator.buildTextConfig`: `"\(voicePrefix)/\(topic.rawValue)/\(stage)"` |
| **Music** | Genre, e.g. `Deep_Dreaming`, `Peaceful_Ambient`, `Solo_Piano`, `Zen_Garden` | `SessionGenerator.buildMusicConfig`; Topic `musicGenre(using:)` or SAD `Solo_Piano` |
| **Nature** | Names from FantasyJourneyManager, e.g. `Gebirgsbach`, `Wald`, `Wasserfall`, `Ozean`, `Hoehle`, `Schneesturm` | `FantasyJourneyManager.Journey.natureSoundName` |
| **Sound** | e.g. `Glockenspiel` | `SessionGenerator.buildSoundConfig` (only for stage `alarm` with Topic OBE) |
| **Noise** | Filename, e.g. `pink.mp3`, `white.mp3` | `NoiseType.audioFileName` (Phase.noise.type) |

**Resolution order (as of 2026 refactoring):** (1) `getSound(byContentId:)`, (2) `SessionContentMapping.soundId(for:)`. There is **no** category-based fallback anymore ("first sound of category"), so each session only loads content from the audio library with a matching `contentId` and not always the same demo track. Failed resolutions are logged (`AWAVELogger.audio.warning`). Without matching Firestore `contentId` entries or SessionContentMapping, only Frequency and optionally Noise are played.

**Mixer display:** In session mode the mixer shows **one name per slot** for the whole session: **Stimme** (slot 0), **Musik** (slot 1), **Naturklänge** (slot 2), **Klänge** (slot 3), plus the separate Waves row for frequency. These labels come from `SessionSlotDisplayNames` and are passed to `PhasePlayer.load(slotDisplayNames:)`; they are not phase-based (e.g. "Einleitung", "Trost"). Phase names remain in `SessionPhase.name` for session tile summaries and accessibility. When no slot display name is provided, the app falls back to the resolved content display name (`Sound.title`) or content ID.

### Sleep Screen and Session Generator (iOS)

- **Hero tile "Besser Schlafen":** A tap opens the **main player** (full player), not the session generator. The full player is presented as `fullScreenCover`.
- **Session generator:** Is only opened via "Neue Sessions generieren" (in the sessions block of the category) or a separate entry point, not via the hero tile. Generated sessions have at least 3 phases (SessionGenerator guarantee).

## Reference: Audio Modules and File Structure

### Audio Modules Overview

| Module     | Use / File |
|-----------|-------------|
| sound     | System sound for Bluetooth click debug; demo of selected voice (`js/main.js`); `AWAVE.html` |
| frequency | `js/generator-frequency-noise.js` |

### Phase Matrix (Guided Meditation vs. Soundscape)

- **Guided meditation:** Text, Music, Nature, Sound, Noise, Frequency (Phase I–IV+).
- **Soundscape:** No text; otherwise same tracks (Music, Nature, Sound, Noise, Frequency).

### Starting a Session (`js/main.js`)

```
startSession() → startPhase(1)
```

`startPhase(phase)` first controls the behaviour of the individual players via the phase duration (fade-in, fade-out, etc.) and then calls:

- `startTextPlayer(phase)`
- `startMusicPlayer(phase)`
- `startNaturePlayer(phase)`
- `startSoundPlayer(phase)`
- `startFrequency(phase)` (Web Audio API, oscillators)
- `startNoise(phase)` (see `generator-frequency-noise.js`)

### Content Change During Live Operation (`js/editor-live.js`)

- **Depending on case,** only restart of the respective player: `startTextPlayer`, `startMusicPlayer`, `startNaturePlayer` …
- **Automatic full session restart** when:
  - Case 1: Session contains only 1 phase
  - Case 2: Timer was changed
  - Case 3: New frequency selected
  - Case 4: New noise selected
  - Case 5: New sound selected  
  → Call: `startSession()`

### Text Player (`js/main.js`)

- Pass `src` to the audio element with `getAudioSrc()`; play with `startTextPlayer(phase)`.
- **Case 1:** `textChoice.mix == "one"` – Length of the audio file determines the phase length.
- **Case 2:** `textChoice.mix == "loop"` – As long as the phase timer has not expired, another audio file is requested with `getAudioSrc()` from `session[phase].text.content` with a delay via `onended`.

### Music Player (`js/main.js`)

- `getAudioSrc()` + `startMusicPlayer(phase)`.
- **Case 1:** `thisPhase === nextPhase` – Same genre → music is not ended at the end of the phase but continued.
- **Case 2:** Otherwise, at the end of the song `onended` calls `startMusicPlayer(phase)` again if the timer has not yet expired.

### Nature Player (`js/main.js`)

- `getAudioSrc()` + `startNaturePlayer(phase)`.
- As long as the phase and session timers have not expired: audio is played in an endless loop with overlap.

### Sound Player (`js/main.js`)

- `getAudioSrc()` + `startSoundPlayer(phase)`.
- Configurations: once at the start of the phase; once at the end of the phase; start and end; at intervals.

### Noise Player (`js/generator-frequency-noise.js`)

- No HTML `<audio>` element; Web Audio API with splitter, bandpass filter, etc.
- Static pass of `src` in `switch (session[phase].noise.content) { … }`.
- Call: `startNoise(phase)`; plays in an endless loop.

### File Path `getAudioSrc()` – Configuration → File Path

1. Read the string from the configured session:  
   `session[phase].text.content` or `music.content`, `nature.content`, `noise.content`, `sound.content`.
2. Convert the string to an object (in `js/content-database.js`):  
   `eval("string")` → DB object `dbItem`.
3. `dbItem` provides e.g. number of songs per music genre, length of text audio files per voice, playback behaviour (`"one"` / `"loop"`).

| Case              | Result |
|-------------------|--------|
| Nature / Noise / Sound | `return "../path/audiofile.mp3";` |
| Text (single file)  | Path including voice name: `../path/voicename_textname.mp3` |
| Text (multiple files) | As above + random number; same number at earliest after 2/3 of pool: `../path/voicename_textname_45.mp3` |
| Music (always multiple)  | Like text (multiple), without voice: `../path/musikgenre_123.mp3` |
